## 1. 개요

ERP에서 다운로드한 근로계약서 현황(xlsx)을 입력 받아, 개인정보를 마스킹 처리한 뒤
구글 스프레드시트에 저장하고, 이를 기반으로

- **사용자**: 본인/조직의 노무비가 적정한 수준인지 확인
- **어드민**: 전체 조직의 노임 추이(시계열 변화)까지 확인

할 수 있도록 하는 웹 애플리케이션이다.

## 2. 기술 스택 (확정)

| 영역 | 기술 | 비고 |
|---|---|---|
| 백엔드/런타임 | Google Apps Script (GAS) | 스프레드시트와 네이티브 연동 |
| 데이터 저장소 | Google Sheets | 별도 DB 없이 시트를 DB로 사용 |
| 프론트엔드 | HTML Service (GAS) + JavaScript | GAS가 서빙하는 웹앱 화면 |
| 차트 | Chart.js (JavaScript) | 노임 추이, 급여 분포 등 시각화 |
| 대시보드(일부) | Google Stitch (외주) | UI 디자인만 외주, 데이터 연동은 자체 개발 |
| 버전관리 | Git + GitHub | 로컬 git init 완료, 원격 저장소는 추후 연결 |
| 로컬 개발 도구 | clasp (Command Line Apps Script Projects) 권장 | 로컬에서 코드 작성 → GAS로 push, git으로 이력 관리 |

> 참고: 초기 요구사항에 언급된 "액션스크립트"는 Adobe Flash ActionScript가 아니라
> Google Apps Script를 의미하는 것으로 확정. "java 차트"도 Java가 아니라
> JavaScript 기반 차트 라이브러리(Chart.js)를 의미하는 것으로 확정.

## 3. 사용자 역할 및 권한

| 역할 | 로그인 화면 | 주요 기능 |
|---|---|---|
| 사용자 (User) | 별도 로그인 화면 | 노무비 적정성 확인만 가능 (본인/소속 범위) |
| 어드민 (Admin) | 별도 로그인 화면 | 적정성 확인 + 노임 추이 확인 + 데이터 전처리(업로드) |

- **로그인 방식 확정**: 사번/개인별 계정이 아니라 **역할별 전사 공용 비밀번호 1개씩** (user 1개, admin 1개). 소스코드에는 절대 하드코딩하지 않고, 비밀번호는 salt+SHA-256 해시로 스크립트 속성(PropertiesService)에 저장 (Users 시트 불필요, git에도 안 올라감).
- 로그인 성공 시 세션 토큰 발급(`Utilities.getUuid()`) → `CacheService`에 6시간 TTL로 저장. 클라이언트는 토큰을 `sessionStorage`에 보관하고 모든 서버 함수 호출에 같이 전달. 모든 데이터/쓰기 함수는 시작 부분에서 토큰의 role을 검증(`requireRole`)한 뒤에만 동작 — 로그인 화면을 우회해 `?page=admin`으로 직접 접속해도 실제 데이터 함수 호출은 차단됨.
- 최초 어드민 비밀번호는 `initializeCredentials()` 함수를 한 번 실행해 랜덤 생성 (반환값으로 1회 노출), 이후 어드민 대시보드의 "비밀번호 관리"에서 admin/user 비밀번호를 각각 변경.
- 구현: [Auth.js](src/Auth.js)
- 사용자별 조회 범위(본인만/부서 전체/전사) 는 아직 논의 필요 — 현재는 역할(user/admin)만 구분하고 조회 범위 제한은 없음 → **토론 필요**

## 4. 데이터 처리 흐름

```
[ERP 근로계약서 현황.xlsx]
        │  (어드민이 업로드)
        ▼
[전처리 단계 (GAS)]
  - 주민등록번호 뒷자리 마스킹 (예: 900101-1******)
  - 성명 마스킹 (예: 홍길동 → 홍OO, 요구사항 상 "최OO" 표기는 예시로 판단)
  - 사번(사원번호)을 고유 매칭 키로 사용
        ▼
[구글 스프레드시트 저장]
  - 원본 시트(비공개/접근제한) / 가공 시트(마스킹 완료, 서비스용) 분리 권장
        ▼
[웹앱 조회 화면]
  - 사용자 화면: 적정성 확인
  - 어드민 화면: 노임 추이 확인
```

- ERP xlsx 실제 컬럼 구조: **`sample(2026-07).xlsx` 확인 완료 (아래 5절 참고)**
- 마스킹은 "원본 → 가공" 1회성 배치 처리로 볼지, 매 업로드마다 누적/갱신할지 결정 필요 → **토론 필요**

## 5. 데이터 모델 (확정 — sample(2026-07).xlsx 기준)

### 5.0 원본 ERP xlsx 구조 (근로계약서 현황)

- 시트명은 매월 타임스탬프가 붙어 달라짐 (예: `▶ 근로계약서 확인_20260716082309`) → **시트명으로 찾지 말고 활성 시트(첫 번째 시트)를 기준으로 처리**
- 1~2행: 헤더(일부 병합), 3행부터 데이터 시작 (사번(C열)이 빈 행은 건너뜀 — 실 데이터에 우연히 공백행이 섞여도 안전하도록 방어적으로 처리)
- 사용하는 컬럼만 기재, 나머지(A,B,F,H,I,J,K,P,Q,R,S,T,U,V,W)는 이번 전처리에서 미사용:

| 열 | 헤더 | 타입/형식 | 용도 |
|---|---|---|---|
| C | 사번 | 문자열 | 이력 추적 키 (PK) |
| D | 성명 | 문자열 | 마스킹 대상 → 성 + 'O'×(이름길이-1) (예: 김하경→김OO) |
| E | 생년월일 | 문자열 `YYMMDD` | 세기 추정 필요 (아래 참고) |
| G | 성별 | 문자열 `남`/`여` | |
| L | 직종 | 문자열 | `안전`, `안전담당자`만 추출 (화이트리스트, 추후 공사/품질/공무 확장 가능) |
| M | 노임구분 | 문자열 `일급`/`월급` | 일급→DAILY(일당제), 월급→MONTHLY(월급제) |
| N | 단가 | 숫자 | 급여액 |
| O | 계약기간-시작일 | 문자열 `YYYY.MM.DD` | 해당 계약서 작성 연월(YYYY.MM) 추출용, 실제 Excel 날짜형이 아닌 텍스트 |

**세기(1900s/2000s) 추정 로직**: 주민번호 뒷자리 코드가 원본에 없으므로, 생년월일 앞 2자리(YY)와 처리 시점의 연도 뒷 2자리를 비교해 추정한다.
`YY <= 기준연도%100 → 20YY, 아니면 19YY` (예: 2026년 기준 YY=04 → 2004년생, YY=96 → 1996년생). 근로 가능 연령대라는 전제하에 성립하는 휴리스틱.

> 주민번호 뒷자리 자체가 ERP 추출 단계에서 이미 빠져 있어(E열은 앞 6자리만), 사양서 초안 8번 "주민번호 뒷자리 마스킹" 요구사항은 소스 데이터 단계에서 이미 충족됨. 전처리에서 실제로 마스킹이 필요한 건 성명(D열)뿐.
> Q열(근로계약서 첨부파일명)은 파일명에 실명이 그대로 노출되어 있어(`김하경_202606.pdf`) 전처리 대상에서 완전히 제외.

구현: [Preprocess.js](src/Preprocess.js), 테스트: [test/preprocess.test.js](test/preprocess.test.js) (`npm test`)

### 5.1 시트: `EmployeeMaster` (직원 마스터, 마스킹 완료본)

| 컬럼 | 설명 | 비고 |
|---|---|---|
| 사번 | 고유 매칭 키 | PK, ERP C열 |
| 성명(마스킹) | 예: 김OO | ERP D열 마스킹, 원본 성명 미저장 |
| 생년월일 | year/month/day | ERP E열 + 세기추정 |
| 성별 | 남/여 | ERP G열 |
| 직종 | 안전/안전담당자 (현재 범위) | ERP L열 |
| 재직상태 | 재직/퇴직 등 | TBD — 원본에 명시 컬럼 없음, 최신 업로드 기준 존재 여부로 판단할지 논의 필요 |

### 5.2 시트: `WageRecords` (급여/계약 이력)

| 컬럼 | 설명 | 비고 |
|---|---|---|
| 사번 | FK | EmployeeMaster 참조, ERP C열 |
| 계약연월 | YYYY.MM | ERP O열(시작일)에서 추출 |
| 노임구분 | DAILY/MONTHLY | ERP M열 |
| 급여(단가) | | ERP N열 |
| 업로드 배치 ID | 어떤 xlsx 업로드분인지 추적 | |

### 5.3 로그인 자격증명 (시트 아님)

Users 시트는 만들지 않음. 역할별 공용 비밀번호(salt+해시)는 스크립트 속성에 저장:
`USER_PASSWORD_SALT`/`USER_PASSWORD_HASH`, `ADMIN_PASSWORD_SALT`/`ADMIN_PASSWORD_HASH`. 3절 참고.
